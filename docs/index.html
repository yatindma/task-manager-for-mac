It is the Windows Task Manager, not a tribute to it

Same seven tabs. Same column order. Same heatmap. Same Ctrl+Shift+Esc muscle memory. Written from scratch in Swift 6 and SwiftUI, reading macOS's own kernel APIs — nothing is emulated and nothing is wrapped.

The heatmap, cell for cell

Windows tints each cell yellow through orange to red as load rises. So does this — the same four-stop gradient, driven by real host_processor_info and proc_pid_rusage samples. You find the busy process by looking, not by reading.

Live graphs on the Windows grid

CPU, memory, disk, network and system-wide GPU, drawn on the same scrolling 60-second grid, in the same colours — CPU blue, memory purple, disk green. Utilisation, speed, processes, threads, handles and uptime, laid out where your eyes already look for them.

End task. End process tree. Suspend. Resume.

Right-click any row and get the menu you expect. End process tree walks the real parent/child graph and takes the children with it. Suspend and resume are SIGSTOP and SIGCONT — the thing Activity Monitor won't give you a button for.

All seven tabs, including the boring ones

Processes, Performance, App history, Startup apps, Users, Details, Services. Startup apps reads your real login items and launch agents. Services reads launchd. The tabs nobody clones are the ones you miss at 2am.

Coming from Activity Monitor

Activity Monitor is a good tool. It is not the tool you have trained on for years. Both read the same kernel, so the numbers agree; the difference is layout, density and vocabulary — which is exactly what muscle memory is made of. If what you want is the Windows Task Manager on a Mac, this is a like-for-like replacement rather than an alternative you have to relearn.

What you get here

The Windows tab order and column order. The load heatmap. Grouped processes with expandable trees. End process tree, suspend and resume in the right-click menu. The 60-second graph grid. Ctrl+Shift+Esc to open it.

What Activity Monitor still does better

It's signed by Apple and already installed. It has the Energy tab and per-process GPU-adjacent power figures. It ships sampling and spindump integration. If you only ever use a Mac, it is probably the right tool and this project has nothing to sell you.

What it can't do, and why

macOS does not expose everything Windows exposes. Where a number cannot be obtained honestly, the app shows zero rather than an estimate dressed up as a measurement. These are the five gaps, in full.

Per-process GPU usage always reads 0%

macOS publishes no public API for per-process GPU utilisation. There is no private workaround this project is willing to ship. The system-wide GPU graph on the Performance tab works correctly; the per-process column does not, and reads 0 for every process.

Root-owned processes need a helper, or they report 0% CPU

Reading CPU time for a process owned by another user requires elevated privileges. The app installs a small setuid helper on demand, behind a single macOS authorisation prompt. Decline it and everything else still works — but roughly 116 of about 620 processes on a typical Mac, including kernel_task and WindowServer, will report 0% CPU.

"Handles" is a file-descriptor count

macOS has no equivalent of the Windows handle. The Handles column shows the open file-descriptor count, which is the closest true analogue. It is not the same number Windows would show, and it isn't pretending to be.

Startup impact is blank

Windows measures the cost of each startup item. macOS measures nothing comparable, and the column has no honest value to display.

Set affinity isn't there

macOS provides no API to pin a process to specific cores. The scheduler does not expose that control to userspace, so the menu item does not exist.

Build it today

There is no signed release yet. Until there is, this is the only way to run it — and it takes one command. You need Xcode's command line tools; there is nothing else to install, because there are no dependencies.

git clone https://github.com/yatindma/task-manager-for-mac.git
cd task-manager-for-mac
./build.sh

Questions

How is this different from Activity Monitor?

Both read the same macOS kernel APIs, so the underlying numbers agree. The difference is presentation. Task Manager for Mac reproduces the Windows 11 layout: the same seven tabs in the same order, the same columns, the yellow-to-red load heatmap Activity Monitor has no equivalent of, grouped process trees, and end process tree / suspend / resume in the right-click menu. It exists for people who use Windows and macOS daily and lose their muscle memory on every switch.

Why does the GPU column show 0 for every process?

Because macOS does not publish a per-process GPU utilisation API — no public framework exposes that number to a normal application. Rather than estimate it, the column reads 0. The system-wide GPU graph on the Performance tab uses a supported API and is accurate.

Why does it ask for my password?

To read CPU usage for processes owned by root and other users, which requires elevated privileges macOS will not grant a sandboxed app. The prompt appears once, when you choose to enable it, and installs a small setuid helper. If you decline, the app works normally except that roughly 116 of about 620 processes on a typical Mac report 0% CPU.

Is the helper safe? What does it actually do?

The helper's only job is to read process accounting data — CPU time, memory and I/O counters — and hand it back to the app. It does not open network connections, write files, or persist a background daemon. It is a small amount of Swift in this repository, it is built from the same source you can read, and you can remove it by deleting it. Nothing here asks you to take that on trust: the entire project is open source with zero third-party dependencies, so there is no code in the build that isn't in the repo.

Does it work on Intel Macs?

Yes. It targets macOS 14 and later on both Apple Silicon and Intel. It is a native Swift 6 and SwiftUI application — not Electron, not a web view, not a port.

Does it slow my Mac down?

A process monitor's cost is its sampling loop. This one polls the kernel on a fixed interval and does no work between samples, the same way Activity Monitor does. It is a native app with no runtime, no bundled browser and no dependencies. If you find a workload where it measurably costs you, that is a bug worth filing.

Does Ctrl+Shift+Esc work?

Yes — the shortcut is the point. It's the reflex that survives the platform switch, so it opens the app on macOS too.

Is it free? What's the licence?

It is free and open source, with no paid tier, no telemetry and no accounts. See the repository for the licence file.
